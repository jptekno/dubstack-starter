## Two approaches to API documentation

DubStack supports two methods for building API documentation:

<CardGroup cols={2}>
  <Card title="OpenAPI Specification" icon="file-code" href="/api-reference/openapi">
    Auto-generate docs from your OpenAPI/Swagger spec. Best for existing APIs.
  </Card>
  <Card title="MDX Components" icon="edit" href="/essentials/markdown">
    Write docs manually using DubStack components. Best for custom documentation.
  </Card>
</CardGroup>

## OpenAPI approach

If you have an OpenAPI specification file, DubStack can automatically generate beautiful, interactive API documentation:

1. **Upload your spec** - Add your `openapi.json` or `openapi.yaml` file
2. **Auto-generate pages** - DubStack creates MDX pages for each endpoint
3. **Interactive playground** - Test endpoints directly in the docs
4. **Code samples** - Automatically generated in multiple languages

<Callout type="success">
**Recommended**: Use OpenAPI for comprehensive APIs with many endpoints. DubStack keeps docs in sync with your spec.
</Callout>

## MDX components approach

For more control, write API documentation manually using DubStack components:

```mdx
---
title: "POST /api/users"
description: "Create a new user"
---

## Create User

Creates a new user account with the provided information.

### Request

<ParamField body="email" type="string" required>
  User's email address. Must be unique.
</ParamField>

<ParamField body="name" type="string" required>
  User's full name.
</ParamField>

### Response

<ResponseField name="id" type="string">
  Unique identifier for the created user.
</ResponseField>

<ResponseField name="created_at" type="timestamp">
  ISO 8601 timestamp of when the user was created.
</ResponseField>

### Example

<CodeGroup>
```bash cURL
curl -X POST 'https://api.example.com/users' \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -d '{"email":"user@example.com","name":"John Doe"}'
```

```javascript Node.js
const response = await fetch('https://api.example.com/users', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${apiKey}`
  },
  body: JSON.stringify({
    email: 'user@example.com',
    name: 'John Doe'
  })
});
const user = await response.json();
```

```python Python
import requests

response = requests.post('https://api.example.com/users',
  headers={'Authorization': f'Bearer {api_key}'},
  json={'email': 'user@example.com', 'name': 'John Doe'}
)
user = response.json()
```
</CodeGroup>
```

## Authentication

All API endpoints in this example use Bearer token authentication:

```bash
curl https://api.example.com/docs \
  -H 'Authorization: Bearer YOUR_API_KEY'
```

Get your API key from the dashboard after signing up.

## Rate limiting

The example API includes rate limiting:

- **FREE tier**: 100 requests/hour
- **PRO tier**: 1,000 requests/hour
- **ENTERPRISE tier**: 10,000 requests/hour

Rate limit headers are included in every response:

```json
{
  "X-RateLimit-Limit": "100",
  "X-RateLimit-Remaining": "95",
  "X-RateLimit-Reset": "1640000000"
}
```

## Error handling

The API uses standard HTTP status codes:

<AccordionGroup>
<Accordion title="200 OK">
  Request succeeded. Response body contains the requested data.
</Accordion>

<Accordion title="201 Created">
  Resource created successfully. Response includes the new resource.
</Accordion>

<Accordion title="400 Bad Request">
  Invalid request. Check the error message for details.

  ```json
  {
    "error": "validation_error",
    "message": "Email is required",
    "field": "email"
  }
  ```
</Accordion>

<Accordion title="401 Unauthorized">
  Authentication failed. Check your API key.

  ```json
  {
    "error": "unauthorized",
    "message": "Invalid or missing API key"
  }
  ```
</Accordion>

<Accordion title="404 Not Found">
  Resource doesn't exist.

  ```json
  {
    "error": "not_found",
    "message": "Documentation page not found"
  }
  ```
</Accordion>

<Accordion title="429 Too Many Requests">
  Rate limit exceeded. Wait before making more requests.

  ```json
  {
    "error": "rate_limit_exceeded",
    "message": "Rate limit exceeded. Try again in 3600 seconds"
  }
  ```
</Accordion>

<Accordion title="500 Internal Server Error">
  Something went wrong on the server. Contact support if this persists.
</Accordion>
</AccordionGroup>

## Best practices

<Steps>
  <Step title="Use descriptive endpoint names">
    Name endpoints after resources, not actions.

    ✅ `GET /users/:id`
    ❌ `GET /getUserById`
  </Step>

  <Step title="Include comprehensive examples">
    Show complete request/response examples in multiple languages.
  </Step>

  <Step title="Document all parameters">
    Describe every parameter, including optional ones and default values.
  </Step>

  <Step title="Show error responses">
    Document all possible error codes and their meanings.
  </Step>

  <Step title="Keep docs in sync">
    Update documentation whenever you change the API.
  </Step>
</Steps>

## Interactive playground

DubStack's API playground lets users test endpoints directly in the docs:

- **Live requests**: Make real API calls from the documentation
- **Auth management**: Save API keys for quick testing
- **Response inspection**: See full responses with syntax highlighting
- **Code generation**: Auto-generate code samples in any language

<Callout type="tip">
The playground is available for both OpenAPI-generated docs and manual MDX documentation.
</Callout>
